# Data Policy

## 데이터 분류

| 데이터 | 저장 위치 | 목적 |
| --- | --- | --- |
| 목록/상세 원본 HTML | `raw_pages` | 파서 오류 재현과 변경 추적 |
| 정규화 공고 | `job_postings` | 검색·상세·매칭의 기준 데이터 |
| 회사 | `companies` | 회사명 중복 제거와 공고 연결 |
| 관찰 이력 | `posting_history` | 공고 변경 및 재수집 확인 |
| 사용자 설정 | JSON 및 preference 테이블 | 설명 가능한 점수 재현 |
| 매칭 결과 | `job_match_results` | 점수·키워드·이유 제공 |
| 내보내기 파일 | `data/*.csv`, `data/*.xlsx` | 실제 지원 검토 |

## 저장 원칙

- 공고 ID가 있으면 최우선 중복키로 사용합니다.
- 공고 ID가 없으면 정규화 URL, 회사·제목·마감일 순으로 대체합니다.
- 외부 데이터는 빈 필드와 형식 오류를 허용하되 성공으로 위장하지 않습니다.
- 상세 수집 실패 원인은 덮어쓰지 않고 확인 가능한 상태로 남깁니다.
- 비밀값은 코드나 JSON에 저장하지 않고 환경변수를 사용합니다.

## 보관 정책

현재 MVP에서는 로컬 SQLite와 export 파일을 Git에서 제외합니다.

운영 단계 권장 기준:

- `raw_pages`: 최근 30일 또는 최근 성공본 N개 보관
- `crawl_runs`: 요약 메타데이터는 장기 보관
- `posting_history`: 변경 이벤트는 장기 보관, 단순 `seen` 이벤트는 주기 압축
- CSV/XLSX: 최근 결과만 유지하거나 날짜별 보관 수 제한

자동 삭제는 백업과 복구 테스트가 준비된 뒤 도입합니다.

## 백업과 복구

1. 크롤링 중이 아닐 때 `data/job_radar.db`를 복사합니다.
2. 복구 전 기존 DB를 별도 이름으로 보존합니다.
3. 복구 후 `jobkorea-health`와 DB 요약을 실행합니다.
4. 스키마 변경 전에는 반드시 백업을 생성합니다.

## 윤리 및 안전

- 대상 사이트의 이용 정책과 robots 지침을 확인합니다.
- 요청 간격을 유지하고 공격적인 병렬 수집을 하지 않습니다.
- 인증, CAPTCHA, 접근 통제를 우회하지 않습니다.
- 수집 차단 또는 rate limit이 감지되면 실행을 중단합니다.
